# 01 · APK 加固总览

## 攻击面（防御者视角）

未加固的 Release APK 常见暴露：

1. **DEX/JAR**：jadx / apktool 可读类名、方法、字符串
2. **资源**：明文 API 地址、功能开关、布局暗示业务
3. **Native `.so`**：未 strip 时符号表泄露算法入口
4. **重打包**：改 smali 后重签安装 → 广告劫持、支付绕过
5. **动态调试**：Frida / ptrace / JDWP 挂钩关键路径

加固目标：让上述每一步都更贵、更吵、更容易被服务端拒绝。

## 心智模型

```text
构建期（R8） → 签名期（v2/v3） → 安装期（渠道/来源）
     ↓                ↓                  ↓
运行期检测（调试/Root） ←→ 服务端裁决（Play Integrity / 自建票据）
```

客户端检测只是信号；**最终决策应在服务端**（封号、降功能、要求重登）。

## 与「壳」的关系

| DIY | 商业壳 |
|-----|--------|
| 可控、可审计、易 CI | 强加密 DEX/SO、反 Dump |
| 保护深度有限 | 黑盒、可能影响启动/兼容 |
| 适合中小 App 基线 | 适合金融、游戏高对抗场景 |

本系列主路径是 **已有 APK 加固**（`pipeline/harden-existing-apk.sh`）：
对编好的包注入运行时壳并重签；深度防护再叠加商业壳。
有工程时，出包前先做 R8。
